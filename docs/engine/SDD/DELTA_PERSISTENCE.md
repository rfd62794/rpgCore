## Overview

Delta-state persistence is the architectural pattern of storing world state changes as a sequence of deltas rather than complete snapshots. This approach provides efficient persistence, deterministic recovery, and clean separation between base world generation and runtime modifications.

## Architectural Precedent

This pattern was previously designed as **ADR 058: Delta-Persistence & World Pillar Formalization** in the DGT Platform. The core insight — world state as a sequence of deltas rather than snapshots — is preserved in the current Procedure specification.

## Pattern Components

### Base World Generation
- **Seed_Zero**: Deterministic seed for world generation
- **Permutation Table**: Hash-based noise generation
- **Base Map**: Procedurally generated world state from seed
- **Deterministic**: Same seed always produces same base world

### Delta State Storage
- **GameState.world_deltas**: Key-value store for persistent changes
- **Delta Format**: `position → {delta_type, timestamp, metadata}`
- **Persistence Strategy**: Save only changes, not entire tile arrays
- **Recovery**: Base world + deltas = complete world state

## Implementation Details

### Delta Schema
```python
@dataclass
class WorldDelta:
    """World state change delta"""
    position: Tuple[int, int]
    original_tile: TileData
    current_tile: TileData
    timestamp: float
    delta_type: str = "tile_change"
    
    def to_dict(self) -> Dict[str, Any]:
        """Convert delta to dictionary for persistence"""
        return {
            "position": self.position,
            "current_tile_type": self.current_tile.tile_type.value,
            "current_walkable": self.current_tile.walkable,
            "current_metadata": self.current_tile.metadata,
            "timestamp": self.timestamp,
            "delta_type": self.delta_type
        }
```

### Persistence Format

## Key Benefits

### Efficiency
- **Storage**: Only save changes, not entire world state
- **Memory**: Base world generated once, deltas applied incrementally
- **Performance**: Fast recovery from base world + deltas

### Determinism
- **Reproducible**: Same seed always produces same base world
- **Testable**: World generation can be tested independently
- **Shareable**: Players can share world seeds

### Clean Separation
- **World Generation**: Pure data provider, no game logic
- **State Management**: Deltas track runtime changes
- **Persistence**: Simple delta serialization

## Integration with ECS

### WorldProcedure Pattern
```python
class WorldProcedure:
    """Procedural world operations with delta tracking"""
    
    def __init__(self, world_seed: str):
        self.world_seed = world_seed
        self.base_world = self._generate_base_world()
        self.deltas: List[WorldDelta] = []
    
    def get_tile_at(self, position: Tuple[int, int]) -> TileData:
        """Get tile data with deltas applied"""
        base_tile = self.base_world.get(position)
        delta_tile = self._find_delta_at(position)
        return delta_tile if delta_tile else base_tile
    
    def apply_delta(self, delta: WorldDelta) -> None:
        """Apply delta and track for persistence"""
        self.deltas.append(delta)
    
    def get_current_world(self) -> Dict[Tuple[int, int], TileData]:
        """Get complete world with all deltas applied"""
        world = self.base_world.copy()
        for delta in self.deltas:
            world[delta.position] = delta.current_tile
        return world
```

## Validation Criteria

### Success Metrics
- **Deterministic World**: Same seed always produces same base world
- **Delta Persistence**: World changes are recorded and recoverable
- **Clean Separation**: World generation only provides data, no game logic
- **No Circular Dependencies**: Clear one-way data flow

### Performance Targets
- **World Generation**: <1 second for 100x100 world
- **Delta Recording**: <1ms per state change
- **Tile Queries**: <0.1ms per tile lookup
- **Memory Usage**: <5MB for world + deltas

## Usage Guidelines

### When to Use Delta Persistence
- **Large Worlds**: When world state is too large for full snapshots
- **Frequent Changes**: When world state changes often
- **Deterministic Generation**: When reproducible world generation is needed
- **Efficient Persistence**: When save/load performance matters

### Implementation Notes
- Base world should be generated once and cached
- Deltas should be applied in chronological order
- Delta format should be versioned for compatibility
- Consider delta compression for long-running games

## Relationship to Other Patterns

### WorldProcedure Pattern
Delta persistence is the foundation of the WorldProcedure pattern. Procedures generate deltas, and deltas provide persistence.

### ECS Architecture
Delta persistence integrates cleanly with ECS by separating world generation (data) from world management (systems).

### Save/Load System
Delta persistence provides an efficient save/load mechanism that works with the ECS component model.
